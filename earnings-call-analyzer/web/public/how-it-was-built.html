How it was built

This tool was built with AI at every step, deliberately, and the process is part of the demonstration. The interesting part is not that AI wrote code. It is the system that kept multiple AI sessions coordinated, evaluated, and honest.

The build, step by step

  1. Strategy before code. Everything started in a single AI chat session with no files and no code: reading the role deeply, identifying that evaluation methodology for LLMs was the differentiator almost no candidate would demonstrate, and designing this tool around that opening. The AI pressure-tested the plan; the decisions were human.
  2. An expensive lesson, turned into architecture. Working over a pay-per-token API, one simple prompt inside an old chat session cost over five dollars, because months of accumulated history rode along silently as context. Bloated context does not just burn money; it degrades output quality. The fix became a discipline: state lives in markdown files, not in conversations. Every work session starts from zero chat history and loads only the files it needs. This is context engineering, practiced rather than claimed.
  3. Parallel AI sessions, coordinated through files. With state in files, multiple sessions ran at once: one on application documents, one on this build. Each session read the shared state files, did its work, and logged results back. When one session locked a decision, the others picked it up from the files. No session ever needed another session's chat history.
  4. An eval-first build. The build session wrote itself a standing brief with a definition of done and guardrails, then executed it in a loop. The pipeline was designed backwards from its evaluation: a golden set of synthetic transcripts with known ground truth, a grounding checker that re-verifies every figure, and a scorecard that must pass before anything ships. It also runs fully offline in a deterministic mode, so the evals are reproducible and free.
  5. Port and parity. The engine was built and evaluated in Python, then ported to TypeScript for Cloudflare's edge runtime. A parity test suite runs the same golden set through the port and requires identical results, so the reference and the deployment cannot drift apart silently.
  6. Deployment like it goes to production. Git push to GitHub, auto-deploy to Cloudflare Pages, API on the Workers runtime at the same origin. Secrets encrypted server-side and never in the client bundle. Password gate on UI and API. A per-IP weekly spend cap enforced through Cloudflare KV. A custom domain. And a stated governance posture: in production this runs on approved infrastructure with data governance and model-risk review.

What was human and what was AI

The strategy, the positioning, the guardrails (no novelty claims, synthetic golden set, no third-party trademarks), and every accept-or-reject decision on AI output: human. The drafting, the code, the port, the tests, and the documentation: AI, working inside the file-based system above, with every session's work logged.

That division of labor is the point being demonstrated. The valuable skill is not using AI. It is directing AI toward institutional-quality output and proving the output deserves trust.
