How it works

Paste a transcript, get an institutional-grade report. In between sits a five-stage engine. Claude is one of those stages. The other four are deterministic code, and they are what makes the output trustworthy enough for an investment desk.

The pipeline

  1. code
    Ingest. The raw file is normalized into clean text and every line gets a stable number: [L1], [L2], [L3]. Nothing analytical happens here. Those line numbers become the audit trail for everything downstream.
  2. code
    Prompt construction. The engine builds a strict instruction set around the numbered transcript: return JSON in an exact schema, cite the line number for every claim, copy figures verbatim from the cited line, and if a claim cannot be cited, omit it entirely. The model is never asked for an open-ended essay.
  3. claude
    Analysis. The only AI stage. Claude reads the numbered transcript and returns structured, cited claims: metrics, guidance, tone, risks, quotes. The call happens server-side; the API key never reaches your browser.
  4. code
    Grounding verification. The engine independently re-checks the model's output: does every citation point to a real line, and does every financial figure in a claim actually appear on its cited line? Anything that fails is flagged on the report in a Grounding Flags section. Flagged, never hidden.
  5. code
    Report rendering. The verified analysis is rendered into a fixed house style defined in a single design-system file, so every report has the same structure, tone rules, and citation format.

Why not just send the file to Claude?

A thin wrapper would produce fluent, confident text with no way to audit it. On an investment desk, one hallucinated figure kills adoption of the whole tool. So the design never asks anyone to trust the model on numbers: the model must show its sources, and plain code re-verifies every figure against those sources after the fact. Trust is structural, not promised.

Where the sentiment analysis lives

The tool has two backends, and the tone read works differently in each:

  • Sample mode (no API key, used for offline demos): a Loughran-McDonald-style lexicon runs in code, counting finance-specific positive and negative terms line by line. Loughran and McDonald (2011) showed that general-purpose sentiment dictionaries misclassify financial text, since words like "liability" read negative in everyday English but neutral in filings. Finance-specific word lists are the classic fix.
  • Live mode (Claude): the model performs the tone read, but the prompt constrains it. Tone claims must cite transcript lines and are framed as a baseline signal, never a verdict. The FinBERT line of research is why a contextual model is trusted over raw word counts: it handles negation and context ("not weak") that dictionary methods miss.

Either way, none of this is presented as novel. Lexicon and transformer-based sentiment are the established baseline in the literature, cited in the methodology document in the repository. The edge of this tool is not the sentiment method. It is that the output is grounded, evaluated, and reproducible.

Python engine, TypeScript edge

The engine was built and evaluated first as a Python package: that version remains the reference implementation and carries the evaluation harness. This web app runs a TypeScript port of the same pipeline on Cloudflare's edge runtime. A parity test suite runs the identical golden set through the ported pipeline and requires identical results, so the two implementations cannot silently drift apart.

How we know it works: the evaluation harness

The pipeline is scored against a golden set of synthetic transcripts with known ground truth (synthetic on purpose: an answer key requires knowing the right answers, and it avoids reproducing copyrighted transcripts). The scorecard measures four things:

  • Extraction accuracy. Were the expected metrics surfaced?
  • Grounding coverage. Is every claim cited?
  • Numeric verification. Does every figure appear on its cited line?
  • Consistency. Do repeated runs agree with each other?

The committed pipeline passes at 100 percent on all four. Evaluation is not an afterthought here; the tool was designed backwards from it.

Access, cost, and data handling

  • The whole site sits behind a password gate, UI and API both.
  • Each visitor has a weekly analysis budget, enforced per IP, to keep demo costs bounded.
  • Transcripts are analyzed in-request and are not stored or logged.
  • This is a demonstration environment. Do not paste confidential or material non-public information.
